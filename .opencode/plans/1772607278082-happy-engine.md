# Neovim Config: Road to Daily Driver

## Structure

**Keep current structure.** `lua/config/` for feature config. `plugin/` for standalone autocmds. No restructuring.

New features: one file per feature in `lua/config/`, added to the require chain.

---

## Phase 1: Daily-Use Essentials

Build one at a time. Evaluate. Tweak. Move on.

### 1. File Picker & Grep

Replace fzf-lua. Want opinionated, fast file finding with frecency. Live grep.

- `<leader>sf` find files, `<leader>sg` live grep
- `vim.ui.select`: use builtin for now
- Skip buffer picker — see if frecency file finding covers it
- **Nix concern:** if the chosen tool has a native binary, need to ensure it works in nix-wrapped nvim (prebuilt download vs cargo build)

**Options:** `fff.nvim` (opinionated, rust backend, frecency, just files & grep)

**File:** `lua/config/picker.lua` (replace current)

### 2. Git Signs / Hunks / Blame

Sign column indicators. Navigate hunks with `]h`/`[h`. Stage/reset hunks. Toggle inline blame. Hunk textobject (`ih`).

Nothing built-in replaces this yet.

**Options:** `gitsigns.nvim` (v2.0, targets nvim 0.11+, de facto standard)

**File:** `lua/config/git.lua`

### 3. Textobjects

Mix of treesitter-based and custom. The goal is specific textobjects, not "install a treesitter textobjects plugin."

**Desired textobjects:**
- `af`/`if` — function definition (treesitter)
- `ac`/`ic` — class/module (treesitter)
- `B` — whole buffer
- `q` — any quotes (nearest quote pair, not just `"` or `'`)
- `o` — ruby block (do..end / `{ }`) (treesitter)
- Markdown: code fences (treesitter)
- `ih` — git hunk (from git signs plugin)

**Note on `gc`:** Built-in comment operator since nvim 0.10. `gcc` comments a line, `gc{motion}` comments a range. Don't override. `gcip` already comments a paragraph. Skip the custom comment-block textobject.

**Options:**
- `nvim-treesitter-textobjects` — treesitter queries for function, class, etc. Still maintained.
- `mini.ai` — framework for defining any `a`/`i` textobject. Has treesitter integration via `MiniAi.gen_spec.treesitter()`. Also enhances builtin textobjects (quotes, brackets). Good home for custom ones like `B` and `q`.
- `nvim-various-textobjs` — 30+ textobjects (indentation, subword, url, etc.), pattern-based. Pick what you need.
- Can mix these. e.g. `mini.ai` as the framework + treesitter queries from `nvim-treesitter-textobjects`.

**File:** `lua/config/textobjects.lua`

### 4. Fast Motions (2-char jump)

Quick cursor movement to any visible location. Want to try 2-char motions (type two chars, land there). May miss flash's "remote" actions but start simple.

**Options:**
- `leap.nvim` — 2-char motions, minimal. `s` to leap.
- `flash.nvim` — search labels on `/` results, treesitter selection, remote actions. More features, more complex.

**File:** `lua/config/motions.lua`

### 5. Motion Repeat

Make `;`/`,` repeat *any* `]`/`[` motion, not just `f`/`t`. Covers: hunk navigation, diagnostic jumps, quickfix, treesitter textobject moves.

The treesitter-textobjects plugin dropped this feature. Needs a standalone solution.

**Options:**
- `demicolon.nvim` — automatically makes all `]`/`[` motions repeatable with `;`/`,`. Also handles `f`/`t`. Clean.
- `vim-remotions` — more general, also works in Vim.
- Manual: use `vim.keymap.set` with a wrapper that stores last motion direction.

**File:** `lua/config/motions.lua` (same file as fast motions, or `plugin/repeat_motions.lua`)

### 6. Treesitter Context

Show current function/class/block at top of buffer while scrolling. **Toggleable** — should be able to turn on/off.

Part of a broader "toggleable UI" pattern (see below).

**Options:** `nvim-treesitter-context`

**File:** add to `lua/config/treesitter.lua`

### 7. Markdown

You work with lots of markdown. Two sub-features:

**a. Wrapping** — Auto-wrap lines in markdown (except code blocks). Remap `j`/`k` to `gj`/`gk` for natural movement over wrapped lines.
- Mostly `vim.opt_local` settings in `ftplugin/markdown.lua`: `wrap`, `linebreak`, `breakindent`, `formatoptions`.

**b. In-buffer rendering** — Render headings, links, code fences, checkboxes, etc. visually in the buffer.
- **Options:** `markview.nvim`, `render-markdown.nvim`

**File:** `ftplugin/markdown.lua` + `lua/config/markdown.lua`

## Cross-Cutting: Toggleable UI

Several features share a pattern: toggle visibility of a UI element. Group them under a consistent keybind prefix.

**Candidates:**
- Treesitter context (on/off)
- Inline git blame (on/off)
- Zen mode (on/off)
- Inlay hints (on/off — already enabled, may want toggle)
- Diagnostics virtual text (on/off)

**Keybind pattern:** `<leader>t` prefix? `yo` (unimpaired-style)? Your call. Key hints plugin would make these discoverable.

---

## Cross-Cutting: Key Hints + Custom Modes

Two related needs:
1. **Key hints** — after pressing a prefix (e.g. `<leader>`), show available continuations. Toggleable, not always on.
2. **Custom modes** — "treewalker mode" where you stay in a submode pressing movement keys repeatedly without a prefix. Also useful for window resizing, git hunk review, etc.

**Options:**
- `which-key.nvim` — key hints + has a "hydra mode" via `show({ loop = true })`. Does both, but hydra support is basic.
- `hydra.nvim` — dedicated submodes plugin. More powerful for custom modes (colored heads, hints, on_enter/on_exit hooks). No key hints for general prefixes though.
- Both together: `which-key` for discoverability, `hydra` for serious submodes.

**File:** `lua/config/keyhints.lua` and/or `lua/config/modes.lua`

## Phase 2: Quality of Life

### 8. Git Merge Conflicts
- Highlight conflict markers visually
- Keybinds to choose ours/theirs/both
- `git jump merge` → populate quickfix (`cgetexpr`)
- **Options:** `git-conflict.nvim`, or DIY with matchadd + quickfix

### 9. Treewalker
- Structured AST movement (move to next sibling, parent, child)
- Wrap in a hydra/submode for rapid traversal
- **Options:** `treewalker.nvim`

### 10. Split / Join
- Toggle single-line ↔ multi-line (arguments, tables, blocks)
- **Options:** `treesj`, `mini.splitjoin`

### 11. Statusline
- Clean it up. Branch, filename, diagnostics, position. Nothing fancy.
- **Options:** `mini.statusline`, `lualine.nvim`, or hand-rolled `vim.opt.statusline`

### 12. Quickfix Filtering
- Filter quickfix list entries without a plugin, or with better UX
- nvim ships `:Cfilter` (`:h cfilter`) — just needs `packadd cfilter`
- **Options:** built-in `cfilter`, `quicker.nvim` (by stevearc/oil author — editable quickfix buffer)

### 13. Quick Wins
- **Jbuilder filetype:** `vim.filetype.add({ extension = { jbuilder = "ruby" } })`
- **Zsh edit-command-line:** filetype detection for the temp file
- **Trailing whitespace:** autocmd on BufWritePre
- **Change case / coercion:** cycle between camelCase, snake_case, kebab-case, etc. `[c]hange [c]ase` prefix idea is nice.
  - **Options:** `vim-abolish` (tpope, also does smart find/replace), or lua alternatives
- **Align text:** `vim-easy-align`, `mini.align`

### 14. Folds
- Already using treesitter foldexpr. Polish UX if needed.
- LSP folding ranges as fallback
- **Options:** built-in + tweaks, `nvim-ufo`

## Implementation Approach

For each feature:
1. Create/edit the config file
2. Add plugin via `vim.pack.add()` if needed
3. Configure keybindings (stock-adjacent)
4. Test in actual editing
5. `nix flake check`

Build one feature at a time. Live with it for a bit. Adjust. Next feature.
