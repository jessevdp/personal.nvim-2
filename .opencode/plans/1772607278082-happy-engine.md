## Cross-Cutting Concerns

These are domains where multiple features overlap. They inform keybind patterns, grouping, and how features relate ‚Äî not implementation order.

**Git** ‚Äî Many features live in the git domain: signs, blame, hunks, conflicts, diff, navigation. They should feel like one cohesive system. Keybinds, commands, and toggles should be consistent across all git features.

**Toggleable UI** ‚Äî Several features are "turn this on/off": treesitter context, inline blame, zen mode, inlay hints, diagnostics virtual text, etc. Needs a consistent keybind pattern so you don't have to remember per-feature toggles.

**Custom modes** ‚Äî Some features want a submode: press a key to enter, stay in that mode, press keys without a prefix, exit cleanly. Should feel first-party: show mode in statusline, clean enter/exit. Treewalker is the clear candidate. There might be others ‚Äî like window resize, diff review ‚Äî worth exploring/validating later as features are built.

**Text operations** ‚Äî Surround, split/join, coercion, swap/reorder are all structured text manipulation. They might share a keybind pattern or at least live nearby. `q` textobject should work in surround too (`csq"` = change surrounding any-quotes to double-quotes) for least surprise.

## Shared UI Spaces

Certain screen areas are written to by multiple features. They're not cross-cutting concerns but shared resources ‚Äî features that touch them should be aware of each other.

**Sign column** ‚Äî Git signs, diagnostic signs, marks, maybe todo highlights. Must not overlap or get too busy. Everything should fit cleanly.
- Git signs: thin bars, staged vs unstaged visually distinct
- Diagnostics: icons/signs (review what's there, clean up)
- Marks: subtle, nice to have (see #38)
- Todo/fixme: probably highlight in-line text, not signcol ‚Äî open question

**Statusline** ‚Äî Multiple features contribute info. Base setup is feature #21. Other features may add components (custom modes, AI, cmdheight 0 extras). Per-filetype overrides needed (Oil, quickfix, undotree, help). Features that touch it should note "Touches: **statusline**."

**Inlay hints / virtual text** ‚Äî Diagnostics, inline blame, maybe LSP type hints. Can compete for visual space on the same line. Priority/toggle matters.

## Navigation Conventions

Bracket motions follow a consistent pattern:
- `]x` ‚Äî next
- `[x` ‚Äî previous
- `]X` ‚Äî last
- `[X` ‚Äî first

Applies to: hunks (`h`/`H`), diagnostics (`d`/`D`), quickfix (`q`/`Q`), and any future bracket motions.

All bracket motions should be repeatable with `;`/`,` (see feature #19).

## Feature Requirements

### 1. File picker & grep
fff.nvim (decided). Replace fzf-lua.

**Requirements:**
- `<leader><leader>` and `<leader>sf` for file search
- `<leader>sg` for live grep
- Default ranking/frecency ‚Äî just use fff.nvim defaults
- Resume last search (`<leader>s.` or similar) ‚Äî **investigate:** does fff.nvim support this?
- Send results to quickfix (new qflist). Nice to have: append to current qflist via separate keybind. ‚Äî **investigate:** fff.nvim recently added qflist support (#216), verify new vs append behavior
- Just files & grep for now. Other pickers (help, symbols, buffers) can come later or from another tool.
- Nix: fff.nvim has a rust binary. Must work in nix-wrapped nvim (prebuilt download or cargo build).

**Possible plugins:** `fff.nvim` (decided)

### 2‚Äì6. Git signs, blame, hunks
Part of: **git**, **toggleable UI** (blame). Touches: **sign column**, **navigation conventions**, **inlay hints/virtual text** (blame).

**Requirements:**
- Thin bar signs in sign column for added/changed/deleted (see: sign column)
- Staged vs unstaged changes should look visually different
- Inline blame, toggleable (see: toggleable UI)
- Navigate hunks: `]h` next, `[h` prev, `]H` last, `[H` first (see: navigation conventions)
- Hunk actions: stage, reset, preview
- `ih` textobject for selecting current hunk
- Open question: `]h` skips staged hunks (semantically correct but confusing mid-review). Maybe "navigate all hunks" vs "navigate unstaged hunks"?

**Possible plugins:** `gitsigns.nvim`

### 7. Textobjects
Relates to: **text operations** (`q` ties into surround).

**Requirements:**
- `aF`/`iF` ‚Äî function definition (`aM`/`iM` alias for method) (treesitter)
- `af`/`if` ‚Äî function call (`am`/`im` alias) (treesitter)
- `ac`/`ic` ‚Äî class/module (treesitter)
- `ao`/`io` ‚Äî block: ruby do..end/`{}`, markdown fenced code block (treesitter)
- `` a` ``/`` i` `` ‚Äî backtick-delimited: matches nearest code_span OR fenced_code_block (treesitter). Use `o` when you specifically want fenced block.
- `ia`/`aa` ‚Äî single argument (treesitter)
- `iA`/`aA` ‚Äî all arguments including parens for `aA` (treesitter)
- `B` ‚Äî whole buffer (select everything). Note: `gqB` already exists as a dedicated LSP format-whole-file mapping ‚Äî that's separate, some formatters need the entire file not line ranges. The `B` textobject is for general use with any operator (`dB`, `yB`, etc.).
- `q` ‚Äî any quotes (nearest surrounding quote pair). Must also work in surround (`csq"` = change surrounding any-quotes to double-quotes).
- `ii`/`ai` ‚Äî indentation level. `ii` = all contiguous lines at same indent (including blank lines between them). `ai` = same + container lines above/below.
- `ih` ‚Äî git hunk (comes from git signs, not this feature)
- Degrade gracefully: if treesitter query isn't available for a language, don't error. Fall back or silently do nothing.
- Best-effort per language for function def vs call distinction.
- Maybe: comment block textobject (contiguous comment lines). Key TBD. `@comment.outer` doesn't work well for multi-line `#` comments in Ruby ‚Äî would need pattern-based approach.
- Maybe: `it`/`at` ‚Äî HTML/XML tag (built-in vim already has this, but could be enhanced)
- Maybe: bracket motions for textobject types ‚Äî `]f`/`[f` next/prev function call, `]F`/`[F` first/last function def, `]m`/`[m` method, etc. Follows navigation conventions (`]X`/`[X` for first/last). Repeatable with `;`/`,`. If the textobjects exist, matching motions are least surprise. Add if easy. Treewalker mode may make these redundant.

**Possible plugins:** `nvim-treesitter-textobjects`, `mini.ai`, `nvim-various-textobjs` (can mix)

### 8. Swap / reorder
Relates to: **text operations**.

**Requirements:**
- Swap function/method definitions with sibling above/below ‚Äî primary use case
- Swap arguments/parameters next/prev ‚Äî nice to have. Best-effort; Ruby's mix of positional and keyword args may cause edge cases. Good enough is fine.
- Could also apply to array elements, hash entries, etc. ‚Äî same "swap sibling" concept but no dedicated treesitter queries for those. If it works, great.
- Keybinds TBD. Text operations keybind scheme is unsettled:
  - `gs` is currently surround, can't reuse
  - `gm` is taken (stock: cursor to horizontal middle)
  - Open question: find a home that doesn't clash
  - Forward/backward swap should feel directional (like n/N or ]/[)

**Possible plugins:** `nvim-treesitter-textobjects` (swap), `iswap.nvim`, `mini.move`

### 9. Surround
Relates to: **text operations**. Shares `q` (any quotes) with textobjects.

**Requirements:**
- Add surround: `gsa{motion}{char}` (e.g. `gsaiw'` = surround inner word with single quotes)
- Delete surround: `gsd{char}`
- Replace surround: `gsr{old}{new}`
- `q` works as a target ‚Äî "any/nearest quotes" (matches textobject `q`)
- Quotes, braces, parens, brackets are the core use case
- Nice to have: tags (`<div>`), function calls. Not used often but nice if the plugin supports it.
- Keybind prefix `gs` ‚Äî currently in use from existing setup. The whole text operations keybind scheme may change. `g..` is a weird namespace ‚Äî might move everything under `<leader>` or find a better pattern. To be refined when text operations (surround, swap, split/join, coercion) are considered together.

### 10. Split / join
Relates to: **text operations**.

**Requirements:**
- Split current node to multi-line
- Join current node to single-line
- Recursive variants: split/join including all children
- Language-aware: e.g. Ruby block `do..end` ‚Üî `{ }` conversion
- Currently happy with treesj behavior. Open to alternatives.
- Keybinds: currently `<leader>c{s,S,j,J}`. May change with text operations keybind scheme.

**Possible plugins:** `treesj`, `mini.splitjoin`

**Possible plugins:** `markview.nvim`, `render-markdown.nvim`

### 17. Copy file reference

**Requirements:**
- Copy relative path (to cwd) ‚Äî the most basic case
- Copy relative path + line number (`file:line` format, rspec-compatible)
- Visual mode: copy with line range (`file:10-25`). Single-line visual = just `file:42`, not `file:42-42`.
- Copy full/absolute path variant
- Maybe: git host URL (permalink to current commit, or link to main branch). GitHub + GitLab support.
- UX: `:CopyReference` command with arguments for variants (e.g. `path`, `absolute`, `url`). Default (no args) = relative path + line. Works in visual mode for line ranges automatically. Tab-completion on arguments. Map the most common variant to a keybind.
- Copies to `+` register (system clipboard)
- Currently have `:CopyRelativePath` ‚Äî upgrade from there

### 18. Fast motions

**Requirements:**
- `s` + two chars to jump anywhere visible ‚Äî any window, bidirectional

- Works in normal mode (overrides stock `s` which is `cl` ‚Äî fine, never used)
- Also in operator-pending and visual mode for least surprise ‚Äî not the primary use case but should just work
- Start simple with 2-char motions. Remote actions and treesitter selection are explore/later (see #39).

**Possible plugins:** `leap.nvim`, `flash.nvim`

### 21. Statusline
Set up the base statusline with what we already have. Other features add to it (see: **statusline** shared space).

**Requirements:**
- Clean, minimal look. Color-coded groups.
- Mode indicator: shortened (N/I/V/VB/VL), color-coded to match theme
- Git: diff stats (+/-/~), color-coded
- Filename with modified/unsaved indicator
- Diagnostic counts (error/warn/info), color-coded
- Line:column, % in file
- Filetype with icon ‚Äî maybe this doubles as a subtle LSP indicator (icon is there = filetype known). Skip explicit LSP server display.

### 24‚Äì26. Git merge conflicts
Part of: **git**. Touches: **navigation conventions**.

resolve.nvim (decided). Handles highlighting, resolution, and diffing.

**Requirements:**
- Semantic highlighting for conflict markers and content sections (ours/theirs/ancestor with colored backgrounds)
- 3-way diff support (diff3 format with ancestor/base)
- Navigate between conflicts via quickfix ‚Äî not `]x`/`[x`. Load conflicts into quickfix and use normal quickfix navigation.

- Resolution keybinds: `<leader>gc` prefix. Register buffer-local keymaps when conflicts detected, clean up when resolved (like LSP keymaps on attach). Use resolve.nvim's `<Plug>` mappings + callbacks.

- Diff views: `<leader>gcd` prefix (base‚Üíours, base‚Üítheirs, both, ours‚Üítheirs). Uses `delta` for rendering.
- Disable resolve.nvim's default keymaps ‚Äî wire up our own.
- Matchit `%` integration ‚Äî jump between markers (resolve.nvim does this, keep it)
- Requires `delta` ‚Äî needs to be available in nix-wrapped nvim

**Possible plugins:** `resolve.nvim` (decided)
