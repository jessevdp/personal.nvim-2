## Cross-Cutting Concerns

These are domains where multiple features overlap. They inform keybind patterns, grouping, and how features relate — not implementation order.

**Git** — Many features live in the git domain: signs, blame, hunks, conflicts, diff, navigation. They should feel like one cohesive system. Keybinds, commands, and toggles should be consistent across all git features.

**Toggleable UI** — Several features are "turn this on/off": treesitter context, inline blame, zen mode, inlay hints, diagnostics virtual text, etc. Needs a consistent keybind pattern so you don't have to remember per-feature toggles.

**Custom modes** — Some features want a submode: press a key to enter, stay in that mode, press keys without a prefix, exit cleanly. Should feel first-party: show mode in statusline, clean enter/exit. Treewalker is the clear candidate. There might be others — like window resize, diff review — worth exploring/validating later as features are built.

**Text operations** — Surround, split/join, coercion, swap/reorder are all structured text manipulation. They might share a keybind pattern or at least live nearby. `q` textobject should work in surround too (`csq"` = change surrounding any-quotes to double-quotes) for least surprise.

## Shared UI Spaces

Certain screen areas are written to by multiple features. They're not cross-cutting concerns but shared resources — features that touch them should be aware of each other.

**Sign column** — Git signs, diagnostic signs, marks, maybe todo highlights. Must not overlap or get too busy. Everything should fit cleanly.
- Git signs: thin bars, staged vs unstaged visually distinct
- Diagnostics: icons/signs (review what's there, clean up)
- Marks: subtle, nice to have (see #38)
- Todo/fixme: probably highlight in-line text, not signcol — open question

**Statusline** — Multiple features contribute info: git, diagnostics, mode, filetype. Covered by feature #21.

**Inlay hints / virtual text** — Diagnostics, inline blame, maybe LSP type hints. Can compete for visual space on the same line. Priority/toggle matters.

## Navigation Conventions

Bracket motions follow a consistent pattern:
- `]x` — next
- `[x` — previous
- `]X` — last
- `[X` — first

Applies to: hunks (`h`/`H`), diagnostics (`d`/`D`), quickfix (`q`/`Q`), and any future bracket motions.

All bracket motions should be repeatable with `;`/`,` (see feature #19).

## Feature Requirements

### 1. File picker & grep
fff.nvim (decided). Replace fzf-lua.

**Requirements:**
- `<leader><leader>` and `<leader>sf` for file search
- `<leader>sg` for live grep
- Default ranking/frecency — just use fff.nvim defaults
- Resume last search (`<leader>s.` or similar) — **investigate:** does fff.nvim support this?
- Send results to quickfix (new qflist). Nice to have: append to current qflist via separate keybind. — **investigate:** fff.nvim recently added qflist support (#216), verify new vs append behavior
- Just files & grep for now. Other pickers (help, symbols, buffers) can come later or from another tool.
- Nix: fff.nvim has a rust binary. Must work in nix-wrapped nvim (prebuilt download or cargo build).

**Possible plugins:** `fff.nvim` (decided)

### 2–6. Git signs, blame, hunks
Part of: **git**, **toggleable UI** (blame). Touches: **sign column**, **navigation conventions**, **inlay hints/virtual text** (blame).

**Requirements:**
- Thin bar signs in sign column for added/changed/deleted (see: sign column)
- Staged vs unstaged changes should look visually different
- Inline blame, toggleable (see: toggleable UI)
- Navigate hunks: `]h` next, `[h` prev, `]H` last, `[H` first (see: navigation conventions)
- Hunk actions: stage, reset, preview
- `ih` textobject for selecting current hunk
- Open question: `]h` skips staged hunks (semantically correct but confusing mid-review). Maybe "navigate all hunks" vs "navigate unstaged hunks"?

**Possible plugins:** `gitsigns.nvim`

### 7. Textobjects
Relates to: **text operations** (`q` ties into surround).

**Requirements:**
- `aF`/`iF` — function definition (`aM`/`iM` alias for method) (treesitter)
- `af`/`if` — function call (`am`/`im` alias) (treesitter)
- `ac`/`ic` — class/module (treesitter)
- `ao`/`io` — block: ruby do..end/`{}`, markdown fenced code block (treesitter)
- `` a` ``/`` i` `` — backtick-delimited: matches nearest code_span OR fenced_code_block (treesitter). Use `o` when you specifically want fenced block.
- `ia`/`aa` — single argument (treesitter)
- `iA`/`aA` — all arguments including parens for `aA` (treesitter)
- `B` — whole buffer (select everything). Note: `gqB` already exists as a dedicated LSP format-whole-file mapping — that's separate, some formatters need the entire file not line ranges. The `B` textobject is for general use with any operator (`dB`, `yB`, etc.).
- `q` — any quotes (nearest surrounding quote pair). Must also work in surround (`csq"` = change surrounding any-quotes to double-quotes).
- `ii`/`ai` — indentation level. `ii` = all contiguous lines at same indent (including blank lines between them). `ai` = same + container lines above/below.
- `ih` — git hunk (comes from git signs, not this feature)
- Degrade gracefully: if treesitter query isn't available for a language, don't error. Fall back or silently do nothing.
- Best-effort per language for function def vs call distinction.
- Maybe: comment block textobject (contiguous comment lines). Key TBD. `@comment.outer` doesn't work well for multi-line `#` comments in Ruby — would need pattern-based approach.
- Maybe: `it`/`at` — HTML/XML tag (built-in vim already has this, but could be enhanced)

**Possible plugins:** `nvim-treesitter-textobjects`, `mini.ai`, `nvim-various-textobjs` (can mix)

### 8. Swap / reorder
Relates to: **text operations**.

**Requirements:**
- Swap function/method definitions with sibling above/below — primary use case
- Swap arguments/parameters next/prev — nice to have. Best-effort; Ruby's mix of positional and keyword args may cause edge cases. Good enough is fine.
- Could also apply to array elements, hash entries, etc. — same "swap sibling" concept but no dedicated treesitter queries for those. If it works, great.
- Keybinds TBD. Text operations keybind scheme is unsettled:
  - `gs` is currently surround, can't reuse
  - `gm` is taken (stock: cursor to horizontal middle)
  - Open question: find a home that doesn't clash
  - Forward/backward swap should feel directional (like n/N or ]/[)

**Possible plugins:** `nvim-treesitter-textobjects` (swap), `iswap.nvim`, `mini.move`

### 9. Surround
Relates to: **text operations**. Shares `q` (any quotes) with textobjects.

**Requirements:**
- Add surround: `gsa{motion}{char}` (e.g. `gsaiw'` = surround inner word with single quotes)
- Delete surround: `gsd{char}`
- Replace surround: `gsr{old}{new}`
- `q` works as a target — "any/nearest quotes" (matches textobject `q`)
- Quotes, braces, parens, brackets are the core use case
- Nice to have: tags (`<div>`), function calls. Not used often but nice if the plugin supports it.
- Keybind prefix `gs` — currently in use from existing setup. The whole text operations keybind scheme may change. `g..` is a weird namespace — might move everything under `<leader>` or find a better pattern. To be refined when text operations (surround, swap, split/join, coercion) are considered together.

### 10. Split / join
Relates to: **text operations**.

**Requirements:**
- Split current node to multi-line
- Join current node to single-line
- Recursive variants: split/join including all children
- Language-aware: e.g. Ruby block `do..end` ↔ `{ }` conversion
- Currently happy with treesj behavior. Open to alternatives.
- Keybinds: currently `<leader>c{s,S,j,J}`. May change with text operations keybind scheme.

**Possible plugins:** `treesj`, `mini.splitjoin`

### 11. Indentation guides

**Requirements:**
- Vertical lines at every indent level
- Current scope's guide highlighted, the rest very subtle
- Show through blank lines (connect guides across empty lines)

**Possible plugins:** `indent-blankline.nvim`, `mini.indentscope`

### 12. Treesitter context
Part of: **toggleable UI**.

**Requirements:**
- Show enclosing scope(s) at top of buffer when scrolled past them
- Generous max lines — useful in deeply nested rspec/rswag files where 10+ lines of context is genuinely needed
- Toggleable — probably off by default since dropbar already shows breadcrumbs. Turn on when deep nesting makes it valuable.
- Maybe: click/jump to the context line
- **investigate:** does it play nice with dropbar? (both occupy top-of-buffer space)

**Possible plugins:** `nvim-treesitter-context`

## Implementation Approach

For each feature:
1. Create/edit the config file
2. `vim.pack.add()` if needed
3. Keybindings (stock-adjacent). Not everything is a keybind — commands are fine.
4. Test in real editing
5. `nix flake check`

One feature at a time. Live with it. Adjust. Next.
